## 一、项目定位与背景

### 1.1 为什么做这个项目

传统电影推荐系统主要走两条路：

- **协同过滤（CF）**：基于"和你相似的人喜欢什么"。冷启动差（新用户/新电影没数据）、无法解释（"为什么推这个"答不上来）、无法响应自然语言意图（用户说"想看烧脑但不晦涩的科幻"完全没法接）。
- **基于内容（Content-based）**：基于电影 metadata（类型、导演、演员）匹配。可解释但语义粒度粗——"烧脑"这种 mood 词无法对应到具体 tag。

RAG 范式可以同时解决这两个问题：
- **语义检索**用向量空间表达"烧脑""治愈""慢节奏"这种模糊概念。
- **生成层**让 LLM 基于召回到的剧情/评论给出推荐理由，天然可解释。
- **结构化过滤**仍然保留（年代、评分等硬约束），不丢精度。

### 1.2 项目范围

- 个人项目，单机原型，不追求工业级吞吐。
- 关注的是 RAG 全栈链路的工程实现与评估方法论，而不是推荐系统的业务最优解。
- 数据规模 ~5 万部电影（豆瓣 + TMDB 中文有效条目过滤后）。

### 1.3 一句话产品形态

用户输入自然语言 query（如"像《盗梦空间》那种烧脑但不要太晦涩的科幻片，2010 年以后的"），系统返回 Top-K 电影列表 + 每部的推荐理由（引用具体剧情/评论片段）。

## 二、整体架构

```
┌──────────────────────────────────────────────────┐
│  User Query (自然语言)                            │
└────────────────────┬─────────────────────────────┘
                     ↓
┌──────────────────────────────────────────────────┐
│  I. Query 结构化解析 (LLM)                        │
│    输出 {semantic_query, filters, similar_to}    │
└────────────────────┬─────────────────────────────┘
                     ↓
┌──────────────────────────────────────────────────┐
│  II. 召回层（三路并行）                           │
│    ├─ Vector Retrieval (bge-large-zh)            │
│    ├─ BM25 (rank_bm25)                           │
│    └─ Metadata Filter (SQL/Chroma where)         │
│    → RRF 融合 → Top-50                           │
└────────────────────┬─────────────────────────────┘
                     ↓
┌──────────────────────────────────────────────────┐
│  III. 精排层 (bge-reranker cross-encoder)        │
│    Top-50 → Top-10                               │
└────────────────────┬─────────────────────────────┘
                     ↓
┌──────────────────────────────────────────────────┐
│  IV. 生成层 (LLM)                                 │
│    基于召回 chunk 生成推荐理由（强制引用）         │
└────────────────────┬─────────────────────────────┘
                     ↓
              推荐列表 + 解释
```

**关键设计原则**：

1. **召回-精排分离**：召回追求 Recall（不漏），精排追求 Precision（不错）。
2. **语义+关键词+结构化三路混合**：单一检索方式都有盲区，三路融合互补。
3. **生成层强制 grounding**：LLM 不能凭参数记忆胡编剧情，必须引用召回内容。

## 三、数据层

### 3.1 数据源

| 来源 | 提供什么 | 为什么需要 |
| --- | --- | --- |
| **TMDB API** | 标准化 metadata：标题、年代、类型、导演、演员、评分、海报 | 结构化字段干净，国际电影覆盖全 |
| **豆瓣公开页面** | 中文剧情简介、中文短评/长评、豆瓣评分 | 中文语义表达，对中国用户更准 |

**为什么用两个源**：TMDB 的中文剧情简介普遍是机翻或者很短，豆瓣的描述质量明显更好；但豆瓣的结构化字段（如导演 ID、演员 ID）不如 TMDB 干净。两边互补。

### 3.2 数据规模与过滤

- **原始拉取**：~15 万条电影记录。
- **过滤后**：~5 万部，过滤规则：
  - 必须有中文剧情简介（长度 ≥ 50 字）
  - 必须有豆瓣评分（≥ 100 人评分，避免冷门样本影响检索质量）
  - 必须有至少一条用户评论
  - 去重（同一部电影豆瓣 ID + TMDB ID 双键去重）

### 3.3 字段 Schema

```python
{
    "movie_id": "tt1375666",         # 主键（用 IMDB ID）
    "title_zh": "盗梦空间",
    "title_en": "Inception",
    "year": 2010,
    "genres": ["科幻", "动作", "悬疑"],
    "director": ["克里斯托弗·诺兰"],
    "cast": ["莱昂纳多·迪卡普里奥", ...],   # Top-10
    "rating_douban": 9.4,
    "rating_count": 1850000,
    "runtime": 148,
    "country": ["美国", "英国"],
    "language": ["英语", "日语"],
    "plot_summary": "...",            # 剧情简介
    "themes_style": "...",            # 主题与风格（评论挖掘 + LLM 总结）
    "review_highlights": "..."        # 高赞评论摘要
}
```

### 3.4 Chunking 策略：结构化切分

**关键决策**：不用 fixed-size chunking（如 LangChain 默认的 RecursiveCharacterTextSplitter），改用**领域结构化 chunking**——每部电影固定切成三块：

| Chunk 类型 | 内容来源 | 用于匹配什么样的 query |
| --- | --- | --- |
| `plot` | 剧情简介 | "讲一个 X 故事的电影"——情节匹配 |
| `themes_style` | LLM 从用户评论中抽取的主题与风格描述 | "烧脑""治愈""慢节奏"——感受/风格匹配 |
| `reviews` | Top-5 高赞短评摘要 | "看完想哭""结尾反转"——观感匹配 |

**为什么这么切**：

- **Fixed-size 的问题**：剧情简介本身长度差异大（50 字 ~ 1500 字），按固定 token 切要么把短剧情切碎、要么把长剧情切散，破坏语义完整性。
- **Semantic chunking 的问题**：需要对每段文本跑 embedding 算相邻段落相似度，5 万部电影 × 多段文本，预处理成本太高，且效果增量有限。
- **结构化 chunking 的优势**：利用领域知识（电影 metadata 本来就有结构），三类 chunk 各自对应明确的 query 意图，召回时还可以**对不同类型的 query 加不同 chunk 类型的权重**。

每个 chunk 携带 metadata：`{movie_id, chunk_type, ...原 metadata}`，方便检索后聚合到电影粒度。

### 3.5 数据预处理细节

- **文本清洗**：去 HTML、统一标点（全角/半角）、去重复字符（"哈哈哈哈哈" → "哈哈"）。
- **themes_style 抽取**：对每部电影的 Top-20 高赞短评走一次 LLM 调用，prompt 模板大致是"提取这些评论中关于电影主题、风格、观感的关键词，输出 3-5 个短语"。这一步是离线一次性预处理，约 5 万次 LLM 调用，用 DeepSeek-V3 控制成本（约 几十块）。
- **海报、预告片**：本项目不处理，纯文本 RAG。多模态留作 future work。

## 四、Embedding 与向量库

### 4.1 Embedding 模型：bge-large-zh

**选型**：BAAI 开源的 `bge-large-zh-v1.5`。

| 参数 | 值 |
| --- | --- |
| 维度 | 1024 |
| 最大 token | 512 |
| 语言 | 中文为主，少量英文混合表现也可 |
| 部署 | 本地 GPU（一张 4090 够用）/ HuggingFace inference endpoint |

**为什么是它**：

1. **中文场景效果好**：C-MTEB 中文检索榜上长期排名靠前，对豆瓣这种中文语料适配好。
2. **开源、本地可部署**：避免依赖商业 API，迭代时不计调用成本。
3. **维度 1024 平衡精度和存储**：5 万 chunk × 3 = 15 万向量，1024 维约 600MB，单机内存完全装得下。

**为什么不用 OpenAI text-embedding-3-large**：

- 维度更高（3072），存储/检索成本翻 3 倍，对 5 万级数据没必要。
- 中文场景对比 bge-large-zh 没有显著优势（MTEB 中文榜 bge 系列长期靠前）。
- 商业 API，迭代时每次重做 embedding 要花钱，且依赖外部服务。

**为什么不微调**：

- 标注数据不够。微调 embedding 需要成对样本（query—relevant_doc），我只有 200 条标注集，远不够。
- bge 已经在大量中文检索任务上训过，开箱效果已经够用。
- 工程优先级：调链路其他环节（混合检索、reranker）收益更高。

### 4.2 向量库：ChromaDB

**选型理由**：

| 维度 | ChromaDB | FAISS | Milvus | Qdrant |
| --- | --- | --- | --- | --- |
| 易用性 | ★★★★★ | ★★★ | ★★ | ★★★★ |
| 元数据过滤 | ★★★★ | 需自己实现 | ★★★★★ | ★★★★★ |
| 部署成本 | 嵌入式，零运维 | 嵌入式 | 需独立服务 | 需独立服务 |
| 5 万规模性能 | 足够 | 最快 | 过度设计 | 足够 |

**选 ChromaDB 的核心理由**：

- 5 万 chunk 是个**小数据集**，FAISS 的极致性能毫无意义，但它做 metadata filter 要自己手写麻烦。
- Milvus / Qdrant 需要独立服务，单机原型用不上。
- ChromaDB 原生支持 metadata where 过滤，可以把"年份 ≥ 2010"这种条件直接下推到检索层。

底层用 HNSW 索引（`hnswlib`），默认参数 `ef_construction=200, M=16` 够用，没特意调。

### 4.3 索引策略

- **每部电影的 3 个 chunk 分别独立索引**，附带 `chunk_type` metadata。
- 检索时可指定 `where={"chunk_type": "themes_style"}` 限定只搜某类 chunk（query 解析后用得到）。
- 检索结果按 `movie_id` 聚合（同一部电影多个 chunk 命中只算一次，分数取 max）。

## 五、召回层（三路并行 + RRF 融合）

### 5.1 为什么要混合检索

单一检索的盲区：

- **纯向量检索**：对"找诺兰导演的电影"这种**精确实体匹配**经常召回不准——embedding 会把"诺兰"和"科恩兄弟""芬奇"等其他导演拉到接近的向量空间。
- **纯 BM25**：对"治愈系电影"这种**纯语义** query 完全失效——治愈这个词本身可能不出现在剧情简介里。
- **纯 metadata filter**：只能处理结构化约束（年代、类型），处理不了语义。

混合检索把三者结合，覆盖三类 query 意图。

### 5.2 三路召回详解

#### 路 1：向量检索

- 用解析出的 `semantic_query`（如"烧脑科幻"）embed 后查 ChromaDB。
- 默认取 Top-50（每路召回数量是个 tunable，详见 5.4）。
- 返回 `[(chunk_id, movie_id, similarity_score), ...]`。

#### 路 2：BM25

- 用 `rank_bm25` 库（标准 BM25Okapi 实现，k1=1.5, b=0.75）。
- 索引建在三类 chunk 的拼接文本上。
- 中文分词用 `jieba`。
- BM25 主要为了捕捉**关键词精确匹配**——导演名、演员名、片名引用（"像《XX》一样的电影"）、特定词汇（"赛博朋克"）。

#### 路 3：Metadata Filter

- 解析出的硬约束直接下推，如 `year >= 2010 AND "科幻" in genres AND rating_douban >= 7.5`。
- 不是真正意义的"排序"，是**候选集裁剪**：把不满足硬约束的全部剔除。
- 在 ChromaDB 里通过 `where` 参数实现，BM25 路也共享同一份候选集。

### 5.3 RRF 融合

**Reciprocal Rank Fusion** 公式：

$$
\text{RRF}(d) = \sum_{i \in \text{retrievers}} \frac{1}{k + \text{rank}_i(d)}
$$

其中：

- `rank_i(d)` 是文档 `d` 在第 `i` 路检索中的排名（1, 2, 3, ...）。
- `k = 60` 是经验常数（Cormack et al. 2009 原论文取值，被广泛沿用）。
- 不在某路结果里的文档，该路贡献为 0。

**为什么用 RRF 而不是加权平均**：

1. **不同检索器的分数尺度不可比**：向量相似度是 [0, 1] 余弦值，BM25 是 [0, ∞) 的 TF-IDF 分。直接加权平均没意义，必须归一化，归一化又涉及一堆超参。
2. **RRF 只看排名不看分数**，天然解决尺度问题。
3. **超参少**：只有一个 k，几乎不用调。
4. **工业界广泛验证**：Elasticsearch、Vespa 这些主流检索引擎都内置了 RRF。

**典型 k 值的影响**：

- k 越小，头部排名差异被放大（rank 1 和 rank 2 分数差距大）。
- k 越大，排名差异被压平，更趋向均匀加权。
- k=60 是被验证过的合理默认值。

### 5.4 各路召回数量

- 每路召回 Top-50，融合后保留 Top-50 进入 reranker。
- 这个数字是个 tradeoff：
  - 太小：召回不全，relevant doc 可能不在 Top-50 里。
  - 太大：reranker 是 cross-encoder，计算量与候选数成正比，太大延迟爆炸。
- 50 是评估集上扫了 {20, 30, 50, 100} 后选的"性能/延迟拐点"。

## 六、Query 结构化解析

### 6.1 为什么需要这一层

用户输入是自然语言，但其中通常混合了：

- **语义意图**："烧脑但不要太晦涩"——必须走向量检索。
- **硬约束**："2010 年以后的""评分 7 分以上"——应该下推到 metadata filter。
- **参照对象**："像《盗梦空间》那种"——可以转化为"基于这部电影的向量做相似检索"。

如果不拆，把整个 query 直接 embed，硬约束会被语义稀释（"2010 年以后"这种话在 embedding 空间没有强信号），导致经常推老片。

### 6.2 Prompt 设计

```text
你是一个电影推荐系统的 query 解析助手。请把用户的自然语言 query 拆解为结构化 JSON：

输出 schema:
{
  "semantic_query": str,          // 用于语义检索的部分，去除硬约束后的纯感受/风格描述
  "filters": {
    "year_min": int | null,
    "year_max": int | null,
    "genres": list[str] | null,   // 类型列表，必须从 [{枚举值}] 中选
    "rating_min": float | null,
    "countries": list[str] | null,
    "directors": list[str] | null
  },
  "similar_to": list[str] | null  // 用户引用的电影名，用于 similar_to 召回
}

约束：
- 不要凭空生成 filter，用户没明说的字段填 null。
- semantic_query 要保留用户的感受性表达，去掉硬约束词。
- 只输出 JSON，不要任何解释。

User query: {user_input}
```

### 6.3 实现细节

- 用 DeepSeek-V3（轻量、便宜、足够），不需要 V3 也跑得动这个任务。
- temperature=0，追求稳定可复现。
- 输出用 JSON mode（DeepSeek 支持），不用做正则解析。
- 解析失败兜底：JSON 解析异常时降级为"semantic_query = 整个用户输入，filters 全空"，保证链路不挂。

### 6.4 `similar_to` 的处理

用户说"像《盗梦空间》那种"时：

1. 在 metadata 库里查到《盗梦空间》的 movie_id。
2. 取它的 `themes_style` chunk 的 embedding，作为额外一路向量检索的 query。
3. 这一路结果也加入 RRF 融合（变成 4 路）。

**为什么这样做**：用户引用一部电影时，通常想要的是"风格相似"，而不是"类型相同"。用 themes_style chunk 而不是 plot chunk 做参照，更贴合用户意图。

## 七、精排层（Reranker）

### 7.1 为什么需要 Reranker

召回层（vector + BM25）的本质是 **bi-encoder**：query 和 doc 分别 encode 成向量再算相似度。优点是快（doc 向量预先算好），缺点是 query 和 doc 之间没有交互，对细粒度相关性建模不够。

**Cross-encoder reranker** 把 query 和 doc 拼起来一起进模型，能建模 token 级别的交互，精度显著高于 bi-encoder。代价是慢——每个 (query, doc) 对都要过一次模型，不能预计算。

所以工业界通用做法：**bi-encoder 召回（快、宽）+ cross-encoder 精排（慢、准）**。

### 7.2 模型选型：bge-reranker-large

- BAAI 开源的 cross-encoder reranker，中文场景效果好。
- 输入 (query, doc)，输出一个 0-1 的相关性分数。
- 推理速度：单卡 4090 上,50 个候选大约 200ms。

**为什么不用 cohere rerank API**：

- 商业 API，迭代成本高。
- 中文场景上 bge-reranker 不输 cohere。
- 本地部署，延迟可控。

### 7.3 精排流程

1. 输入：召回的 Top-50 chunks（已聚合到 movie 粒度，每部电影取最高分的 chunk 作为代表）。
2. 对每个 (query, chunk_text) 对跑 reranker，得到相关性分数。
3. 按分数降序，输出 Top-10。

### 7.4 一个坑：长 chunk 截断

- bge-reranker 最大 token 512。
- 部分 plot chunk 较长，超过会被截断，可能丢掉关键信息。
- 处理：超长 chunk 在精排时换成"标题 + 主要类型 + 前 400 token"，避免无脑截断。

## 八、生成层（基于检索的解释）

### 8.1 为什么生成解释

只输出列表用户不知道"为什么推这个"——尤其是 mood-based query。给每部电影生成一段"为什么这部符合你的需求"，能：

- 提升信任感。
- 让用户判断推荐合不合适，不用每部都查一遍。
- 暴露推荐错误（如果解释牵强，用户能立刻发现）。

### 8.2 Prompt 设计与 Grounding

```text
你是一个电影推荐助手。用户的需求是：{user_query}

候选电影：《{title}》
相关信息：
- 剧情：{plot_chunk}
- 主题风格：{themes_chunk}
- 用户评论：{reviews_chunk}

请用 2-3 句话说明这部电影为什么符合用户需求，要求：
1. 必须基于上面提供的信息，不要编造剧情、演员、导演等事实。
2. 必须明确指出符合用户哪个需求点。
3. 如果某个需求点没法从信息中确认，直接说"具体是否符合 XX 需要观看后判断"，不要瞎编。
```

**Grounding 的关键**：

- prompt 里**只给召回到的 chunk**，不给 LLM 凭参数记忆补全的机会。
- 明确要求"不要编造"，并给出"承认不确定"的兜底话术。
- temperature=0.3（不全 0 是为了语言自然，但不让它太发散）。

### 8.3 防幻觉的额外措施

LLM 在电影领域最容易瞎编的场景：

- 把演员张冠李戴（"由汤姆·汉克斯主演"——但他根本没演过）。
- 编虚假剧情情节。
- 给错评分、错年代。

**应对**：

- 演员、导演、年代、评分这类**结构化字段**直接从 metadata 里渲染到结果展示里，**不让 LLM 生成**。LLM 只生成"主观推荐理由"部分。
- 用户看到的格式：

```
《盗梦空间》(2010) 9.4/10  导演：克里斯托弗·诺兰
推荐理由（由 AI 生成）：这部电影的多层梦境结构和潜意识规则带来强烈的烧脑感，
但叙事主线清晰，不至于让你迷失，符合你"烧脑但不晦涩"的需求。
```

- 结构化部分由系统填，AI 部分受 grounding 约束，事实错误概率显著降低。

## 九、评估体系

### 9.1 评估集构建

- **规模**：200 条 (query, relevant_movie_ids) 标注对。
- **Query 来源**：
  - 1/3 自己写（典型 mood-based、参照式 query）。
  - 1/3 改写自豆瓣"片单"标题（"治愈系日影""硬核科幻 Top10"等真实用户表达）。
  - 1/3 从 LLM 生成（用 prompt 让它扮演不同人群提出推荐需求）。
- **Relevant labels**：每条 query 标 5-15 部 relevant 电影。标注方法是先用 v1 系统跑出 Top-50，人工过滤后再补充遗漏。

**坦诚说明**：这个标注集**有偏差**——
- 来自 v1 系统的候选可能有 selection bias（v1 漏掉的电影更难进入标注集）。
- 200 条规模偏小，统计置信区间宽。
- 缓解：每次大版本迭代后重新审视标注集，补充新发现的 relevant 项。

### 9.2 评估指标

| 指标 | 计算方式 | 衡量什么 |
| --- | --- | --- |
| **Recall@10** | Top-10 里 relevant 电影数 / 总 relevant 数 | 召回完整性 |
| **NDCG@10** | 折扣累积增益，越靠前的 relevant 权重越大 | 排序质量 |
| **MRR** | 1 / 第一个 relevant 出现的位置 | 用户最快多久看到合适推荐 |
| **LLM-as-judge** | LLM 评估推荐解释是否与事实一致 | 生成层幻觉率 |

### 9.3 三个版本的指标演进

| 版本 | 改动 | Recall@10 | NDCG@10 | 说明 |
| --- | --- | --- | --- | --- |
| **v1** | 纯向量检索 (bge-large-zh) | 0.52 | 0.48 | 基线，问题：精确实体匹配差、硬约束失灵 |
| **v2** | 加入 BM25 + metadata filter，RRF 融合 | 0.64 | 0.59 | 关键词类 query 大幅提升，"导演名"召回从 30% 到 80%+ |
| **v3** | 加入 bge-reranker 精排 | 0.73 | 0.71 | Top-10 排序质量显著改善，NDCG 涨幅最大 |

**每一步的归因**：

- v1 → v2：BM25 解决了实体匹配，metadata filter 解决了硬约束。bad case 分析里"年代不对"和"导演召回不全"几乎消失。
- v2 → v3：召回已经够全（Recall@50 ≈ 0.85），瓶颈在排序——relevant 电影在 Top-50 里但排到 20-30 位。reranker 把它们重新顶到 Top-10。NDCG 涨幅 (0.59 → 0.71) > Recall 涨幅 (0.64 → 0.73)，符合 reranker"提排序不提召回"的理论预期。

### 9.4 LLM-as-judge 评估生成质量

- 用 Claude 3.5 Sonnet 作为 judge（避免和生成层用同一模型造成偏差）。
- Prompt 模板：给 judge 看用户 query + 推荐电影的 metadata + AI 生成的解释，让它判断 (1) 解释是否与已知事实一致 (2) 是否回应了 user query。
- 抽样 50 条解释跑 judge，事实一致率 ~92%，主要错误是"过度推断剧情细节"——chunk 里没说但 LLM 强行补充。
- 改进：把 prompt 里的 grounding 约束加强后涨到 ~96%。

## 十、关键设计决策与权衡

> 这一节是面试拷打区。每个决策都准备好"为什么是这个，不是那个"。

### 10.1 为什么是召回-精排两阶段，不是端到端检索

- 端到端 cross-encoder 检索（每个 query 对全库 5 万 chunk 跑一遍 cross-encoder）：精度最高，但 query 时延几十秒，**完全不可用**。
- 召回-精排：召回阶段用 bi-encoder 快速缩到 50 个候选，精排只对 50 个跑 cross-encoder，时延能压到 1 秒内。
- 这是 IR 领域的标准范式（Google、Bing、各大搜索引擎都是这么做的）。

### 10.2 为什么用 RRF 不用 weighted sum

见 5.3。补充一点：**Weighted sum 需要为每路检索器调权重**，权重又会随 query 类型变化（"导演名"query 应该重 BM25，"mood"query 应该重 vector）。要么搞动态权重（复杂），要么固定权重（次优）。RRF 完全绕开这个问题。

### 10.3 为什么 chunking 是结构化的，不是 fixed-size 或 semantic

见 3.4。补充一点：结构化 chunking 还有一个隐藏收益——**chunk 类型本身可以作为 query 路由信号**。Query 是 "找烧脑科幻" → 应该主搜 themes_style chunk；query 是 "讲二战的电影" → 应该主搜 plot chunk。这是 fixed-size 给不出的。

### 10.4 为什么不微调 embedding

见 4.1。核心是**数据不够 + ROI 太低**：

- 标注集 200 条远不够微调。
- bge-large-zh 已经在大规模中文检索数据上训过，开箱足够。
- 同样的工程精力，加 reranker 比微调 embedding 收益高一个数量级（NDCG +0.12）。

工程师该把精力花在 ROI 最高的地方，不是花在"听起来最炫"的地方。

### 10.5 为什么用 DeepSeek，不用 GPT-4o/Claude

- **Query 解析、解释生成**这种任务，DeepSeek-V3 完全够用，错误率没比 GPT-4o 显著差。
- **成本差异**：DeepSeek 比 GPT-4o 便宜约 10 倍。5 万部电影做 themes_style 预处理 + 200 条评估集多轮迭代，差价可观。
- **Judge 用 Claude**：评估时换一个家族的模型避免"自卖自夸"偏差。

### 10.6 为什么不上多模态（海报、预告片）

- 个人项目，多模态 embedding 模型（CLIP/Chinese-CLIP）部署、对齐都是新工作量。
- 文本信号已经足够覆盖大部分 query。
- 留作 future work。

## 十一、局限与改进方向

**诚实暴露的局限**：

1. **评估集规模偏小**：200 条 query 的结论统计稳健性有限。如果做产品级，至少要 1000+ 条，分场景分层。
2. **冷启动新片仍弱**：新片缺少用户评论，themes_style chunk 质量差。可以从相似旧片继承 themes 作为兜底。
3. **个性化缺失**：当前是 query-based 推荐，没考虑用户历史偏好。加上用户画像后是另一套系统（CF + RAG 融合）。
4. **没考虑多样性**：Top-10 全是相似电影会降低用户体验。MMR 这种多样性重排没做。
5. **不支持多轮对话**：用户改需求要重发 query。要做成对话式推荐，需要把对话历史也纳入 query 解析。

**真要继续做的改进顺序**：

1. **多样性重排（MMR）**：1 天工作量，明显提升体验。
2. **对话式 query refinement**：让用户在结果里说"不要这种类型"，系统迭代。
3. **个性化**：用户历史 → 用户兴趣 embedding → 加权 RRF 第 5 路。
4. **多模态**：海报 embedding 作为额外路。

## 十二、高频深挖问题与回答

### A. 检索基础

#### Q12.1 Embedding 是怎么训练出来的？bge 用了什么方法？

> bge 系列用的是**对比学习 + 多阶段训练**：
>
> 1. 预训练阶段：在大规模无标注中文语料上做 MLM（masked language modeling）等任务，学到通用语义表示。
> 2. 对比学习阶段：用 (query, positive_doc, negative_docs) 三元组训练，让 query 和 positive 在向量空间靠近、与 negatives 远离。
> 3. 任务微调阶段：在 MS MARCO 中文版、DuReader 等检索任务上微调。
>
> 关键技术：**hard negative mining**（不是随便采负样本，而是采"看着像但其实不相关"的硬负样本），让模型学会精细区分。

#### Q12.2 余弦相似度、欧氏距离、点积，向量检索用哪个？

> 取决于 embedding 是否归一化。
>
> - bge 输出已经做了 L2 归一化，此时**余弦 ≡ 点积 ≡ 1 - 欧氏距离²/2**，三者排序结果完全等价。
> - 没归一化的 embedding 用点积会被向量长度污染，应该用余弦。
>
> ChromaDB 默认用余弦，我没改。

#### Q12.3 HNSW 索引的原理是什么？

> Hierarchical Navigable Small World，分层图索引：
>
> - 把向量构建成多层图，每层节点数指数级减少，最顶层只有几个节点。
> - 检索时从顶层开始贪心搜最近邻，逐层向下精化。
> - 类似跳表（skip list）的思想，但用在图上。
>
> 核心参数：
> - `M`：每个节点的连接数，越大召回越好但内存越多。
> - `ef_construction`：建索引时的搜索宽度。
> - `ef_search`：查询时的搜索宽度，运行时可调（精度/速度的 tradeoff）。
>
> 5 万级数据完全没必要调，默认值就行。百万级以上才需要细调。

#### Q12.4 BM25 公式是什么？为什么比 TF-IDF 好？

> BM25 公式（针对单个 query term `q_i` 在 doc `D` 中）：
>
> $$ \text{score}(D, q_i) = \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{\text{avgdl}})} $$
>
> 关键改进相比 TF-IDF：
>
> 1. **TF 饱和**：词频对分数的贡献有上限（`k_1` 控制），不像 TF-IDF 是线性的。一个词出现 10 次不应该比出现 5 次重要 2 倍。
> 2. **长度归一化**：长文档天然词频高，BM25 用 `b` 参数惩罚长文档，避免长文档霸榜。
>
> 常用参数：`k_1=1.5, b=0.75`。

### B. RAG 系统设计

#### Q12.5 Chunk size 怎么定？

> 没有银弹，权衡：
>
> - **太小**：上下文不全，embedding 学不到完整语义，检索精度下降。
> - **太大**：单个 chunk 包含多个语义点，embedding 被稀释（averaging effect），不同 query 都会匹配但都不准。
>
> 经验：
> - 通用文档：500-800 token，overlap 50-100 token。
> - 我这个项目用结构化 chunking，每个 chunk 本身就是一个完整语义单元（剧情/主题/评论），自然长度 100-600 token，不强行切。
>
> 真要做 fixed-size 调优，方法是：固定其他变量，扫 chunk_size ∈ {200, 400, 600, 800} 看 Recall@K 指标，选拐点。

#### Q12.6 Chunk overlap 为什么需要？

> 防止语义边界被切碎。比如"主角发现了真相。原来 X 就是凶手。" 切在第一句末尾，第二个 chunk 就完全没有上下文。
>
> Overlap 让相邻 chunk 共享一部分边界文本，保证关键信息不会因为切分位置丢失。代价是存储和检索冗余。
>
> 结构化 chunking 不用 overlap，因为切分边界本身是语义边界（剧情→主题→评论）。

#### Q12.7 RAG 怎么处理多跳问题？比如"诺兰最新的科幻片是什么"

> 单次召回的 RAG 处理不好多跳，因为需要先查"诺兰的所有科幻片"，再筛"最新"。
>
> 几种方法：
>
> 1. **Query 重写**：让 LLM 把多跳 query 拆成单跳，分步检索。我这个项目的 query 解析层就有一点这个意思（把 "similar_to" 拆出来单独召回）。
> 2. **Self-RAG / IRCoT**：让 LLM 在生成过程中决定要不要再检索一次。
> 3. **结构化 + 非结构化混合**：把可结构化的部分（导演=诺兰、类型=科幻、按年份排序）走 SQL，剩余语义走向量。我这个项目用的就是这套思路。
>
> 真正复杂的多跳 query（"和 A 风格相似但比 B 更冷门的电影"）当前 RAG 范式还是处理不好，要 agentic RAG 才行。

#### Q12.8 怎么评估 RAG 的"是不是真用了检索"，还是模型在凭参数记忆答？

> 几种方法：
>
> 1. **故意给错检索结果**，看 LLM 是被误导（说明真在用检索）还是无视（说明在用参数知识）。
> 2. **要求引用具体 chunk**，看引用是否真实来自检索结果。
> 3. **抠 attention**（如果模型开放）：看生成时 attention 是否分布在检索内容上。
>
> 我这个项目用的是方法 2 ——prompt 强制引用 + 后处理验证（解释里提到的剧情元素必须能在召回 chunk 里找到对应片段）。

### C. 工程实现

#### Q12.9 5 万部电影的 embedding 怎么离线算？花多久？

> 单卡 4090 上，bge-large-zh batch_size=32, 5 万 × 3 = 15 万 chunk，大约 30-40 分钟跑完。
>
> 工程注意：
> - 用 `sentence-transformers` 的 `encode(batch_size=32, normalize_embeddings=True)`。
> - 不要单条 encode，吞吐量差几十倍。
> - 长 chunk 先截断到 512 token，避免 OOM。

#### Q12.10 系统怎么部署、QPS 是多少？

> 单机原型，没追求高 QPS。链路串行：query 解析 (500ms) + 召回 (50ms) + 精排 (200ms) + 生成 (1500ms) ≈ 2.5 秒/请求。
>
> 真要工业部署，瓶颈在 LLM 调用：
> - Query 解析能并发（每个请求独立）。
> - 生成层 Top-10 解释可以**并发请求 LLM**，不是串行。
> - Reranker 走本地 GPU batch 推理。
>
> 真追求高 QPS 还要考虑：召回结果缓存（同 query 缓存 5 分钟）、Embedding 缓存等。

#### Q12.11 如果数据从 5 万扩到 500 万，架构哪里需要改？

> 主要三处：
>
> 1. **向量库**：ChromaDB 单机扛 500 万有点紧，换 Milvus 或 Qdrant，走分布式。
> 2. **Embedding 存储**：500 万 × 3 × 1024 维 = 60GB，必须分片。
> 3. **BM25 索引**：rank_bm25 是内存索引，扛不住 500 万。换 Elasticsearch 或 Vespa。
>
> 召回-精排架构本身不用变，这是和数据量无关的范式。

### D. 灵魂拷问

#### Q12.12 这个项目和"基于内容的推荐"有什么本质区别？听起来差不多

> 关键区别在**输入侧的灵活性**和**输出侧的可解释性**：
>
> - 传统 content-based：输入必须是结构化（用户选 tag），输出只能是列表。
> - RAG 推荐：输入是自然语言（mood + 硬约束 + 引用），输出有推荐理由。
>
> 底层都是基于内容相似度，但 RAG 把"内容相似度"从 tag 匹配升级成"语义匹配 + 关键词匹配 + 结构化过滤"的混合体，能处理 tag 表达不出的需求。

#### Q12.13 你的 0.73 Recall@10 跟工业系统比怎么样？是高还是低

> 老实说，**这个数字本身缺乏可比性**，因为：
>
> 1. 我的评估集是自建的，和公开 benchmark 不可比。
> 2. 工业系统的"relevant"定义是用户点击/观看/打分，我的是人工标注，口径不同。
>
> 在我的评估集上，0.52 → 0.73 是一个**相对提升 40%**，能反映出"混合检索 + 精排"这套范式的工程价值。绝对数值不是关键，关键是**有完整的归因链路**（每个版本改了什么、涨了多少、为什么）。

#### Q12.14 如果让你重做一遍，会怎么改

> 三点：
>
> 1. **更早搭评估集**：我 v1 跑出来很久后才意识到没有量化评估、靠人眼看结果迭代效率太低。如果重来第一周就先标 50 条评估集，让每次改动都有数字反馈。
> 2. **更早做 bad case 分析**：从 v1 到 v2 我花了挺多时间在"我觉得 BM25 应该加上"这种直觉驱动，后来才系统化地把 v1 的 bad case 拉出来分类，发现 60% 都是"导演名/演员名召回不准"。这种分析应该在第一版结果出来就做。
> 3. **更早考虑链路成本**：第一版没限制 LLM 调用，跑评估集都要花十几块。如果重来会先把所有 LLM 调用 cache 起来（同 query 同模型同 prompt 走 cache）。
