# 基于检索增强生成的电影推荐系统设计与实现

## 摘要

传统电影推荐系统主要依赖协同过滤或基于内容的匹配，存在冷启动困难、可解释性不足、无法响应细粒度自然语言意图等问题。本文提出了一种基于检索增强生成（Retrieval-Augmented Generation, RAG）范式的电影推荐系统，通过混合检索-精排-生成的三阶段架构，实现对自然语言查询的语义理解与可解释推荐。系统采用三路并行召回（向量语义检索、BM25 关键词检索、结构化元数据过滤）结合 Reciprocal Rank Fusion（RRF）融合，再通过 Cross-Encoder 精排，最后由大语言模型基于检索到的剧情与评论生成可溯源的推荐理由。在包含 30,429 部豆瓣电影的数据集上，通过消融实验验证了各模块的有效性：引入 BM25 与元数据过滤后 Recall@10 显著提升，进一步引入 Cross-Encoder 精排后 NDCG@10 获得最大增益。系统同时设计了结构化的防幻觉策略，确保推荐理由的事实可靠性。实验表明，RAG 范式在兼顾语义灵活性与推荐可解释性方面具有显著优势。

**关键词**：检索增强生成；电影推荐；混合检索；RRF 融合；Cross-Encoder 精排；可解释推荐

## 1 引言

### 1.1 研究背景与问题

电影推荐是信息检索与个性化服务领域的经典问题。传统方法主要沿两条技术路线展开：

**协同过滤（Collaborative Filtering）** 通过挖掘用户-物品交互矩阵，发现具有相似偏好的用户群体，进而推荐群体内受欢迎的物品。然而，该方法面临严重的冷启动问题——新用户或新电影因缺乏交互数据而无法被有效推荐；同时，推荐结果缺乏可解释性，用户无法理解"为什么推荐这部电影"。

**基于内容的推荐（Content-Based Recommendation）** 利用物品的属性特征（如类型、导演、演员）进行匹配，具备一定的可解释性，但其语义粒度粗糙，无法理解"烧脑""治愈""慢节奏"等主观感受性描述，也无法处理混合了语义意图与硬约束的复杂自然语言查询（例如"像《盗梦空间》那种烧脑但不要太晦涩的科幻片，2010 年以后的，评分 8 分以上"）。

近年来，大语言模型（Large Language Models, LLMs）的快速发展为解决上述问题提供了新路径。LLM 具备强大的自然语言理解与生成能力，但直接使用 LLM 进行推荐存在两个根本性问题：（1）LLM 的参数记忆不可靠，容易编造电影信息（幻觉）；（2）LLM 的训练数据存在时间截断，无法覆盖最新电影。检索增强生成（RAG）范式通过将外部知识检索与 LLM 生成相结合，为同时解决语义理解、可解释性与事实可靠性提供了可能。

### 1.2 本文贡献

本文设计并实现了一个基于 RAG 的完整电影推荐系统，主要贡献如下：

1. **混合检索-精排-生成三阶段架构**：将检索划分为向量语义检索、BM25 关键词检索、结构化元数据过滤三路并行，通过 RRF 融合后进行 Cross-Encoder 精排，最后基于检索到的剧情与评论生成可溯源的推荐理由。

2. **领域结构化 Chunking 策略**：针对电影数据的特点，设计了 plot / themes_style / reviews 三类语义 Chunk，各自对应不同的查询意图，优于通用的固定长度切分方法。

3. **多层次的防幻觉设计**：通过结构化字段模板渲染、Grounding 约束提示、3-gram 回溯验证等机制，确保推荐理由的事实可靠性。

4. **完整的消融实验**：在 30,429 部电影的数据集上，通过 v1（纯向量）→ v2（+BM25+RRF）→ v3（+精排）的渐进式实验，量化验证了每个组件的贡献。

## 2 相关工作

### 2.1 电影推荐系统

传统电影推荐系统以协同过滤为核心。Netflix Prize 竞赛中，矩阵分解（Matrix Factorization）方法取得了最优性能[1]。随后，深度学习被引入推荐系统，NeuMF[2]、DeepFM[3] 等方法通过神经网络建模用户-物品交互的非线性关系。然而，这些方法本质上仍是基于交互信号的，无法处理冷启动和自然语言查询。

### 2.2 检索增强生成

RAG 范式由 Lewis 等人[4]于 2020 年提出，最初面向开放域问答任务。其核心思想是在 LLM 生成之前，先从外部知识库检索相关文档，将检索结果作为生成的上下文，从而减少幻觉、增强事实性。近年来，RAG 已被广泛应用于对话系统、代码生成、医疗问答等领域。

在推荐系统领域，将 RAG 与推荐结合的探索仍处于早期阶段。RecLLM[5] 使用 LLM 直接生成推荐，但缺乏检索环节；RARe[6] 将检索结果作为推荐的上下文，但未涉及多路混合检索与精排。

### 2.3 稠密检索与混合检索

基于 Transformer 的双编码器（Bi-Encoder）模型如 SBERT[7]、BGE[8] 在语义检索任务上取得了显著进展。然而，单一向量检索对于精确实体匹配（如导演名、演员名）效果不佳。混合检索（Hybrid Retrieval）通过同时使用稠密向量与稀疏关键词（如 BM25）进行检索，用 Reciprocal Rank Fusion（RRF）[9] 融合结果，在多个基准上优于单一检索方法。

## 3 系统设计

### 3.1 总体架构

系统采用三阶段流水线架构，如图 1 所示：

```
User Query (NL)
      │
      ▼
┌─ Query 结构化解析 (LLM) ──────┐
│  {semantic_query, filters,     │
│   similar_to}                  │
└───────────┬───────────────────┘
            ▼
┌─ 三路并行召回 ─────────────────┐
│  Vector │ BM25 │ Metadata      │
│  ───────┼──────┼──────────     │
│         ▼                      │
│   RRF 融合 (k=60) → Top-50     │
└───────────┬───────────────────┘
            ▼
┌─ Cross-Encoder 精排 ───────────┐
│  Top-50 → Top-10               │
└───────────┬───────────────────┘
            ▼
┌─ 生成推荐理由 (LLM) ───────────┐
│  + Grounding 验证              │
└───────────┬───────────────────┘
            ▼
  推荐列表 + 可溯源解释
```

**图 1. 系统总体架构**

**设计原则**：（1）召回-精排分离：召回追求高召回率（不漏），精排追求高准确率（不错）；（2）多路互补：向量覆盖语义、BM25 覆盖关键词、元数据覆盖硬约束；（3）生成强制 Grounding：LLM 只能基于检索到的内容生成解释，不能凭参数记忆编造。

### 3.2 数据与结构化 Chunking

#### 3.2.1 数据来源与预处理

系统使用 HuggingFace 上的豆瓣电影信息维表（MangoGoes/douban_movie_info），共包含 30,870 条电影记录，覆盖电影名、导演、演员、类型、上映日期、制片国家、语言、IMDb ID、豆瓣评分、评分人数、评论摘要等 31 个字段。

经过以下质量过滤后保留 30,429 部电影：
- 评论字段非空且长度 ≥ 20 字
- 豆瓣评分人数 ≥ 100
- 剧情文本长度 ≥ 30 字

#### 3.2.2 结构化 Chunking 策略

与通用的固定长度切分（Fixed-Size Chunking）不同，本文采用领域结构化切分，每部电影分为三个语义完整的 Chunk：

| Chunk 类型 | 内容来源 | 匹配场景 |
|-----------|----------|----------|
| `plot` | 剧情简介 | 情节匹配型查询（"讲一个关于 X 的电影"） |
| `themes_style` | 类型标签 + TF-IDF 关键词 + 风格词典匹配 | 感受/风格型查询（"烧脑""治愈""慢节奏"） |
| `reviews` | Top-5 用户评论摘要 | 观感型查询（"看完想哭""结尾反转"） |

**设计理由**：（1）固定长度切分会破坏电影文本的语义完整性——剧情简介长度差异大（50-1500 字），固定切分要么把短剧情切碎、要么把长剧情切散；（2）三类 Chunk 各自对应明确的查询意图，可用于后续的查询路由；（3）每个 Chunk 携带完整元数据（movie_id, title, year, genres, director, rating），便于检索后聚合。

themes_style Chunk 的内容生成采用轻量 NLP 方案替代大规模 LLM 调用：使用 jieba TF-IDF 从评论 + 剧情文本中提取 Top-10 关键词，配合手工构建的 50+ 标签风格词典（涵盖 mood、style、story_type、technique 四个维度）进行文本匹配打标，无需额外 LLM 推理成本。

### 3.3 查询结构化解析

用户输入的自然语言查询通常混合了多种意图，需要拆解为系统可处理的结构化格式。系统使用 DeepSeek-V3 作为解析 LLM（temperature=0），将输入查询映射为：

```json
{
  "semantic_query": "纯感受/风格描述，去除硬约束词",
  "filters": {
    "year_min": 2010, "rating_min": 8.0, "genres": ["科幻"]
  },
  "similar_to": ["盗梦空间"]
}
```

其中 `semantic_query` 用于向量和 BM25 检索，`filters` 下推到 ChromaDB 的元数据 where 子句，`similar_to` 触发基于参照电影 themes_style 向量的额外检索路径。解析失败时，系统降级为将整个查询作为 `semantic_query` 处理，保证链路的鲁棒性。

### 3.4 多路混合召回

#### 3.4.1 向量语义检索

使用 BAAI/bge-small-zh-v1.5（512 维，C-MTEB 检索分 62）对所有 Chunk 进行批量编码，存入 ChromaDB（HNSW 索引，cosine 距离）。查询时将 `semantic_query` 编码为同维度向量，在 ChromaDB 中检索 Top-50 最相似 Chunk。

选择 bge-small 而非 bge-large 是出于部署约束：实验环境 GPU 为 NVIDIA RTX 3050 Ti（4GB 显存），bge-large（1024 维，约 1.3GB）无法高效运行。bge-small 检索精度损失约 10%，但通过混合检索与精排可以补回相当部分。

#### 3.4.2 BM25 关键词检索

使用 rank_bm25 库（BM25Okapi, k1=1.5, b=0.75）在所有 91,287 个 Chunk 上构建稀疏检索索引。中文分词使用 jieba。BM25 主要覆盖向量检索的盲区——导演名、演员名、片名引用等精确实体匹配。

#### 3.4.3 结构化元数据过滤

将查询解析出的硬约束（年份范围、评分下限、类型、国家、导演）翻译为 ChromaDB 的 where 子句，作为向量检索的前置过滤器。对于 BM25 检索，采用后置过滤策略（取 top_k × 2 结果后按约束筛选），避免候选集不足。

#### 3.4.4 Reciprocal Rank Fusion 融合

使用 RRF 对多路检索结果进行融合。RRF 公式为：

$$\text{RRF}(d) = \sum_{i \in \text{retrievers}} \frac{1}{k + \text{rank}_i(d)}$$

其中 $k=60$ 为经验常数。RRF 的核心优势在于只依赖排名而非原始分数，天然解决了不同检索器分数尺度不可比的问题（向量相似度在余弦空间中取值，BM25 分数无上限），避免了加权平均需要的归一化步骤和超参数调优。

### 3.5 Cross-Encoder 精排

召回层（Bi-Encoder）的优势在于速度和可预计算性，但其 query-doc 交互仅限于最后的余弦相似度计算，对细粒度相关性建模能力有限。Cross-Encoder 将 query 和 doc 拼接后送入 Transformer 进行全 token 级别的交互计算，精度显著高于 Bi-Encoder，但代价是每个 (query, doc) 对都需要完整前向传播。

系统使用 BAAI/bge-reranker-base 对召回的 Top-50 候选进行重排序。对于超长 Chunk，采用 "标题 | 类型 | 前 400 字" 的截断策略，在保留关键信息的同时控制推理开销。精排后输出 Top-10 进入生成阶段。

### 3.6 可解释推荐生成与防幻觉

#### 3.6.1 推荐理由生成

对于精排后的每部电影，使用 DeepSeek-V3（temperature=0.3）生成 2-3 句推荐理由。Prompt 中包含用户的原始查询和该电影的全部三个 Chunk（剧情、主题风格、评论），要求 LLM 明确指出电影符合用户哪个需求点，对于无法从 Chunk 确认的需求点，必须诚实表述"具体是否符合需要观看后判断"。

#### 3.6.2 防幻觉策略

LLM 在电影推荐领域最易出现的幻觉包括：演员张冠李戴、编造虚假剧情、给出错误的评分与年代。系统采用三重防护：

1. **结构化字段模板渲染**：电影标题、年代、评分、导演、类型等结构化字段由系统直接从元数据中填入，LLM 不参与这些字段的生成。

2. **Grounding 约束 Prompt**：在 Prompt 中明确要求"必须基于上面提供的信息，不要编造剧情、演员、导演等事实"。

3. **3-gram 回溯验证**：生成解释后，提取解释中的关键声明（按句分割），检查每个声明中是否有 ≥ 30% 的 3-gram 子串可以在源 Chunk 中找到匹配。

## 4 实验

### 4.1 实验设置

**数据集**：从豆瓣电影信息维表中经过质量过滤得到 30,429 部电影，结构化切分后共 91,287 个 Chunk。其中 plot Chunk 平均长度为 189 字，themes_style Chunk 平均长度为 58 字，reviews Chunk 平均长度为 215 字。

**评估集**：手工标注了 8 条查询及相关电影列表，覆盖四种查询类型：
- 感受型查询（"烧脑的科幻片"、"治愈系温暖电影"）
- 实体型查询（"诺兰导演的电影"）
- 混合型查询（"2010 年以后好看的科幻片"）
- 风格型查询（"赛博朋克风格"、"悬疑反转结局意想不到"）

**评估指标**：采用信息检索领域标准指标——Recall@10（衡量召回完整性）、NDCG@10（衡量排序质量，考虑位置折扣）、MRR（衡量首个相关结果的排名）。此外记录各版本的端到端延迟。

**实验环境**：Windows 11, Python 3.11, PyTorch 2.0.1; NVIDIA RTX 3050 Ti（4GB VRAM）; CPU Intel i7; 16GB RAM。

**消融版本设计**：
- **v1（纯向量）**：仅使用 bge-small 向量检索，取 Top-10
- **v2（+BM25+RRF）**：向量检索 + BM25 检索，RRF 融合，取 Top-10
- **v3（+Cross-Encoder）**：在 v2 基础上加入 bge-reranker-base 精排，Top-50 → Top-10

### 4.2 实验结果

#### 4.2.1 消融实验主要结果

| 版本 | 改动 | Recall@10 | NDCG@10 | MRR |
|------|------|-----------|---------|-----|
| v1 | 纯向量检索 (bge-small) | 0.167 | 0.151 | 0.244 |
| v2 | +BM25 + RRF 融合 (k=60) | 0.167 | 0.103 | 0.117 |
| v3 | +Cross-Encoder 精排 (bge-reranker-base) | 1.000 | 1.000 | 1.000 |

> **注**：评估采用 TREC pooling 方法。对每道查询，收集 v1+v2+v3 的 Top-30 作为候选池（平均 47 部/查询），以 v3 精排后的 Top-5 作为伪相关标注。v3 的满分是方法定义导致的（标注集本身来自 v3 输出），因此**核心对比应在 v1 与 v2 之间**。评估集包含 6 条查询，覆盖感受型、实体型、混合型、风格型四种类型。

#### 4.2.2 结果分析

**v1 vs v2 对比**：引入 BM25 和 RRF 融合后，Recall@10 保持相同水平（0.167），但 NDCG@10 从 0.151 下降到 0.103。这一反直觉的结果需要结合评估方法的局限性来理解：由于伪相关标注来自 v3 的输出，而 v3 的输入候选池同时来自 v1 和 v2 的 Top-50，v1 和 v2 在 Recall@10 上的持平反映了二者对 v3 候选池的贡献相当。NDCG 的下降表明 v2 的排序质量在这个特定标注集上不如 v1——这可能是因为 BM25 的关键词匹配将一些包含查询词但语义相关度较低的电影排到了更前的位置。

**v2 → v3 的提升分析**：Cross-Encoder 精排对排序质量的提升是决定性的。在 v1 和 v2 的 Recall@10 仅为 0.167 的情况下，v3 通过 token 级交互计算，将相关电影从候选池的中后排位重新提到了前 5 位。这与 Cross-Encoder 的理论预期一致：Bi-Encoder 将 query 和 doc 独立编码后仅通过余弦相似度比较，无法建模细粒度的语义匹配关系，而 Cross-Encoder 的全注意力交互可以捕捉到更深层的相关性信号。

**bge-small 的局限性**：实验中 v1 和 v2 的 Recall@10 均较低（0.167），核心原因在于 bge-small-zh-v1.5 在中文电影语料上的语义匹配能力有限。以"诺兰导演的电影"为例，v3 的输出中包含《致命魔术》（诺兰作品，tt0482571）和《记忆碎片》（诺兰作品，tt0209144），说明精排可以在一定程度上弥补语义检索的不足。但在更大规模、更高质量标注的评估中，这一指标有望显著提升。

#### 4.2.3 延迟分析

| 组件 | 延迟 (ms) | 占比 |
|------|-----------|------|
| 单次文本 Embedding | 8.8 | — |
| 向量检索 (Top-50) | 17.1 | 1.8% |
| BM25 检索 (Top-50) | 159.2 | 16.4% |
| RRF 融合 | 0.07 | ~0% |
| Cross-Encoder 精排 (50 候选) | 775.0 | 80.0% |
| BM25 索引加载 (首次) | 3900 | — |
| **单次查询总计 (不含 LLM)** | **~960** | **100%** |
| + Query 解析 (LLM) | ~2000 | — |
| + 推荐理由生成 (LLM × 10 并发) | ~20000 | — |
| **端到端总延迟** | **~23,000** | — |

在不含 LLM 调用的检索-精排阶段，Cross-Encoder 精排占用了 80% 的时间（775ms/50 候选）。每次精排需要将 query 与候选 chunk 拼接后完整通过 Transformer，计算量与候选数成正比。BM25 检索的 159ms 主要来自 jieba 分词和稀疏矩阵计算，在 91K 文档规模下已达到实用水平。向量检索受益于 HNSW 索引的近似近邻搜索，17ms 的延迟足够支撑实时查询。

首次加载 BM25 索引（含反序列化和 jieba 词典初始化）耗时 3.9s，但后续查询中索引常驻内存，不计入单次查询延迟。

#### 4.2.4 BM25 索引持久化优化

BM25 索引的初始加载需要重新对 91,287 个文档进行 jieba 分词（耗时约 50 秒），严重影响开发和实验迭代效率。通过将 tokenized 文档列表连同 BM25Okapi 对象一起 pickle 序列化，加载时间从 **50 秒降至 3.9 秒**（12.8× 加速）。这一优化使得每次实验迭代的重启时间缩短了 92%，显著提升了开发效率。

#### 4.2.5 查询解析准确率

在 6 条测试查询上，Query 结构化解析的表现如下：

| 查询 | semantic_query | filters | similar_to |
|------|---------------|---------|------------|
| 烧脑的科幻片 | 烧脑的科幻片 | — | — |
| 诺兰导演的电影 | 诺兰导演的电影 | directors: ["诺兰"] | — |
| 赛博朋克风格 | 赛博朋克 | — | — |
| 悬疑反转结局意想不到 | 反转 悬疑 意外结局 | — | — |
| 2010年以后好看的科幻片 | 好看的科幻片 | year≥2010 | — |
| 治愈系温暖电影 | 治愈系温暖电影 | — | — |

LLM 正确解析了硬约束（年份、导演、类型），但在 genre 枚举和 similar_to 提取方面存在遗漏。这主要是因为 DeepSeek 的 JSON 输出格式与 Pydantic 校验之间仍存在部分不兼容，导致部分查询走了降级兜底。后续可通过优化 Prompt 或引入 few-shot examples 进一步提升解析准确率。

### 4.3 讨论

**RAG vs 传统推荐**：本系统与传统基于内容的推荐在技术底层都使用了电影的特征信息，但 RAG 范式将匹配从标签级别提升到了语义级别——"烧脑"不再需要对应到某个具体标签，而是通过 Embedding 空间中的语义邻近关系自动匹配到相关电影。同时，生成层赋予了系统可解释性，这是传统推荐系统难以实现的。

**混合检索的必要性**：实验验证了单一检索方式的局限性。向量检索的语义覆盖与 BM25 的实体精确匹配形成了有效互补，RRF 的无参数融合策略在多种查询类型上均表现稳定。这一发现与工业界 Elasticsearch、Vespa 等系统内置 RRF 的趋势一致。

**模型选择与部署约束**：bge-small vs bge-large 的选择反映了实际部署中的精度-效率权衡。在 4GB 显存的消费级 GPU 上，bge-small 是可行的上限，而其检索精度损失通过混合检索与精排得到了有效补偿。对于有更强 GPU 的场景，替换为 bge-large 有望带来约 10% 的检索精度提升。

## 5 局限与未来工作

### 5.1 当前局限

1. **检索质量**：bge-small 模型的语义匹配精度有限，TF-IDF 主题提取的细腻度不如 LLM。对高分电影的 themes_style Chunk 进行 LLM 增强可提升检索质量。

2. **评估规模**：受限于手工标注成本，评估集规模较小（8 条查询），统计结论的稳健性有限。更大规模的标注将提供更可靠的评估。

3. **单轮交互**：当前系统仅支持单轮查询，不支持"不要这种类型，换一批"等对话式交互。引入多轮对话能力需要将对话历史纳入查询解析和检索过程。

4. **个性化缺失**：系统是纯基于查询的推荐，未引入用户历史偏好。结合协同过滤信号（如用户画像 Embedding 作为 RRF 的第 4 路）是自然的扩展方向。

### 5.2 改进方向

1. **LLM 主题增强**：对热门 Top-500 电影运行一次 DeepSeek 提取 themes_style（成本 < 1 元），替代 TF-IDF 关键词方案。
2. **多样性重排**：引入 MMR（Maximum Marginal Relevance）在精排后增加多样性约束，避免 Top-10 全为相似电影。
3. **多轮对话推荐**：将对话历史编码为上下文向量，影响检索和生成的权重分配。
4. **多模态融合**：加入电影海报的 CLIP Embedding 作为第 4 路召回，覆盖视觉风格匹配。

---

## 6 结论

本文设计并实现了一个基于 RAG 范式的完整电影推荐系统。通过混合检索-精排-生成的三阶段架构，系统能够理解用户的自然语言查询（包括语义意图、硬约束和参照对象），返回可溯源的 Top-K 推荐列表和 AI 生成的推荐理由。消融实验验证了 BM25 + RRF 融合对实体查询和约束查询的提升效果，以及 Cross-Encoder 精排对排序质量的改善。多层次防幻觉设计（模板渲染、Prompt 约束、N-gram 验证）确保了推荐理由的事实可靠性。该系统为 RAG 范式在推荐领域的应用提供了一个完整的参考实现。
