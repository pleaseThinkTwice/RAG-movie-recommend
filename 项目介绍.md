# RAG 电影推荐系统 — 项目介绍

## 一句话概述

一个基于 RAG（Retrieval-Augmented Generation）的电影推荐系统。用户输入自然语言，系统返回匹配的电影列表 + AI 生成的个性化推荐理由。

## 解决了什么问题

传统电影推荐有三大痛点：

| 痛点 | 传统方案 | 本系统 |
|------|----------|--------|
| **自然语言输入** | 只能选标签/类型，无法表达"烧脑但不晦涩" | 直接输入自然语言，LLM 自动拆解意图 |
| **可解释性** | 只出列表，"为什么推这个？"答不上来 | 每部电影附带 AI 生成理由，引用剧情/评论 |
| **语义匹配** | 标签匹配粒度粗，"治愈""慢节奏"无法对应 | 向量语义检索 + 关键词 + 结构化过滤三路融合 |

## 核心架构

```
用户输入 "像《盗梦空间》那种烧脑科幻，2010年以后，8分以上"
        │
        ▼
┌─ Query 结构化解析 (DeepSeek-V3) ──────────────────┐
│  拆解为: semantic_query="烧脑科幻"                  │
│          filters={year≥2010, rating≥8}              │
│          similar_to=["盗梦空间"]                     │
└──────────────────────┬────────────────────────────┘
                       ▼
┌─ 三路并行召回 ─────────────────────────────────────┐
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ 向量检索  │  │ BM25 检索│  │  元数据过滤      │  │
│  │(bge-small)│  │(jieba)   │  │(year,rating,etc)│  │
│  └────┬─────┘  └────┬─────┘  └────────┬─────────┘  │
│       └──────────────┼───────────────┘             │
│                      ▼                              │
│              RRF 融合 (k=60) → Top-50               │
└──────────────────────┬────────────────────────────┘
                       ▼
┌─ 精排 (bge-reranker-base Cross-Encoder) ───────────┐
│  50 候选 → 逐对计算 query-doc 相关性 → Top-10       │
└──────────────────────┬────────────────────────────┘
                       ▼
┌─ 生成推荐理由 (DeepSeek-V3 + Grounding) ───────────┐
│  基于召回的剧情/评论生成解释，禁止编造               │
│  结构化字段(标题/评分/导演) 模板渲染，LLM 不碰        │
└──────────────────────┬────────────────────────────┘
                       ▼
              《盗梦空间》(2010) 9.4/10
              推荐理由: 多层梦境结构带来强烈烧脑感，
              但诺兰清晰的主线确保不会迷失...
```

## 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| **Embedding** | BAAI/bge-small-zh-v1.5 | 512维，中文语义检索，4GB VRAM 可跑 |
| **向量库** | ChromaDB | HNSW 索引，支持元数据 where 过滤 |
| **关键词检索** | BM25 (rank_bm25 + jieba) | k1=1.5, b=0.75 |
| **融合算法** | Reciprocal Rank Fusion | k=60，只看排名不看分数 |
| **精排** | BAAI/bge-reranker-base | Cross-Encoder 逐对打分 |
| **LLM** | DeepSeek-V3 | Query 解析 + 推荐理由生成 |
| **数据** | HuggingFace 豆瓣数据集 | 30,429 部电影，91,287 个 chunk |
| **语言** | Python 3.11 | Pydantic v2, sentence-transformers |

## 数据与 Chunking

- **数据来源**: HuggingFace `MangoGoes/douban_movie_info`（豆瓣电影信息维表）
- **数据规模**: 30,870 条原始 → 30,429 条通过质量过滤
- **结构化 Chunking**: 每部电影切为 3 个语义 chunk：

| Chunk 类型 | 内容 | 匹配场景 |
|-----------|------|----------|
| `plot` | 剧情简介 | "讲一个 X 故事的" |
| `themes_style` | 类型 + TF-IDF 关键词 + 风格标签 | "烧脑""治愈""慢节奏" |
| `reviews` | 用户评论摘要 (Top-5) | "看完想哭""结尾反转" |

## 关键设计决策

### 为什么是 RRF 而非加权平均？
不同检索器的分数尺度不可比（向量相似度 [-1,1]，BM25 分数 [0,∞)），RRF 只看排名，天然绕过归一化问题。

### 为什么结构化 Chunking 而非固定长度？
固定长度会把剧情切碎、长剧情切散。plot/themes/reviews 三类各自是完整语义单元，且 chunk 类型可作为 query 路由信号。

### 为什么结构化字段不让 LLM 生成？
LLM 在电影领域极易编造事实（张冠李戴演员、虚构剧情）。标题/评分/导演/类型由模板直出，LLM 只生成主观推荐理由，大幅降低幻觉风险。

### 为什么用 bge-small 而非 bge-large？
RTX 3050 Ti 仅 4GB 显存，bge-large (1.3GB) 无法高效运行。bge-small (440MB) 检索精度损失约 10%，但三路混合召回 + 精排可补回相当部分。

## 防幻觉设计

1. **结构化字段模板渲染**: 标题、年份、评分、导演、类型 → 系统直接填入，LLM 不生成
2. **Grounding 约束 Prompt**: "必须基于提供的信息，不要编造" + "无法确认时请说明"
3. **3-gram 回溯验证**: 生成后检查解释中的声明能否在检索 chunk 中找到对应文本
4. **温度控制**: 解释生成 temp=0.3（不全为 0 是为了语言自然度），Query 解析 temp=0

## 评估体系

- **评估集**: 20 条手工标注 query（覆盖 mood / 实体 / 混合 / similar-to 四类）
- **指标**: Recall@K, NDCG@K, MRR
- **LLM-as-judge**: Claude 评估解释的事实一致性（不同模型家族避免偏差）
- **版本追踪**: v1(纯向量) → v2(+BM25+元数据) → v3(+精排) 指标演进

## 快速运行

```bash
# 1. 安装依赖
pip install chromadb sentence-transformers rank-bm25 jieba openai pydantic pyyaml datasets

# 2. 设置 API Key
export DEEPSEEK_API_KEY=your_key

# 3. 数据预处理 (~2min)
python -c "from src.data.pipeline import run_pipeline; run_pipeline()"
python -c "from src.chunking.chunker import run_chunking; run_chunking()"

# 4. 构建索引 (~30min, 仅首次)
python -m src.cli.build_index

# 5. 查询推荐
python -m src.cli.recommend "想看治愈系日本动画电影"

# 6. 运行评估
python -m src.cli.evaluate

# 7. 单元测试
pytest tests/ -v  # 22 passed
```

## 已知局限

1. **检索质量**: bge-small 精度有限，热门电影的主题提取依赖 TF-IDF 而非 LLM
2. **冷启动**: 缺乏用户评论的电影 themes_style chunk 质量下降
3. **无个性化**: 当前是纯 query-based，未引入用户历史偏好
4. **无多样性控制**: Top-10 可能全是相似电影，未做 MMR 重排
5. **不支持多轮对话**: 每次查询独立，不能"不要这种类型，换一批"

## 真要继续做的改进

1. **热门电影 LLM 主题增强**: 对 Top-500 电影跑 DeepSeek 提取 themes_style（成本 < 1 元）
2. **多样性重排 (MMR)**: 1 天工作量，明显提升体验
3. **对话式推荐**: 用户说"不要这种"→ 系统迭代
4. **用户画像**: 历史偏好 embedding → RRF 第 4 路加权
5. **多模态**: 海报 CLIP embedding 作为额外召回路径
